
### CRM Web App — Key Design Decisions

React 18 + TypeScript SPA at `apps/crm/web`. Vite build, React Router v6, Tailwind CSS, React Query (TanStack v5). **Shell:** Fixed left sidebar + top bar; sidebar nav items are role-adaptive, derived from `AuthContext.permissions` (no API call per render). **Location model:** Global `LocationContext` (`activeLocationId: string | null`; `null` = All Locations for marketing roles); `apiFetch` injects `location_id` only when `activeLocationId` is a non-null, non-empty string — never injected as `null`. Location switch auto-invalidates all location-scoped React Query keys (zero reload). **State (3 contexts only):** `AuthContext` (user, role, `permissions: Set<string>` derived from `ROLE_PERMISSIONS[role]` at login), `LocationContext`, `NotificationContext` (fed by SSE); React Query for all server state; no Zustand/Redux. **SSE:** `NotificationProvider` at `AppShell` level; subscribes to `user:{userId}` always + `location:{locationId}:*` when `activeLocationId` is set; omits location channel entirely when `isAllLocations === true` — Notification Service has no special handling for `locations:[]`; `lastSeq` stored in component state (not localStorage). **RBAC:** Client-side `ROLE_PERMISSIONS` mirrors `@ortho/auth-middleware`; `<RequirePermission redirect="/403">` for route gates; component-level `<RequirePermission>` renders `null`/fallback only (never redirects); `<RequireLocation>` for entity-scoped actions. **`must_change_password`:** `AppShell` redirects all routes to `/settings/password`; only `/settings/password` + `GET /identity/me` allowed. **Module architecture:** Feature-module SPA; `src/modules/{leads,inbox,analytics,campaigns,sequences,automation,audience,templates,referrals,reports,import,settings}/`; each module is `React.lazy`-loaded (code-split at module boundary). **Platform UI embedding:** `/sequences`, `/automation`, `/audience`, `/templates` are full-page routes that mount the respective `@platform/*` package, passing `apiClient` (from `src/lib/api/`) + `canPublish` + `locationId` as props; packages call their own service APIs directly, never via CRM API Gateway. **API clients:** `src/lib/api/` — `crm-gateway.ts` (leads, pipeline, conversations, campaigns, referrals, import, reporting), `identity.ts`, `media.ts`, `notification.ts`, `template.ts`, `sequence.ts`, `audience.ts`, `automation.ts`. **Coordinator metrics:** "My Performance" sidebar item for `call_center_agent` links directly to `/analytics/coordinators`; Reporting Service pins `coordinator_id` to JWT `sub`; frontend resolves coordinator display names by calling `GET /identity/users/:id` in parallel with user's own JWT (Reporting Service returns `coordinator_id` only — it has no service-level access to Identity Service). **Role default redirects:** agent/coordinator → `/leads`; marketing_staff → `/analytics`; marketing_manager → `/leads`; super_admin → `/settings`. **Inbox:** Top-level nav item (separate from Leads); two-panel layout (conversation list + thread); AI draft strip above composer. **Leads:** Two-panel (queue list + detail); detail has Overview/Activity/Messages/Notes tabs; Quick Actions always visible. **Analytics:** Sub-tabs — Channel Performance (default), Funnel, Locations, Coordinators, Reports; all sourced from Reporting Service via standard JWT auth (frontend never calls Analytics Service directly). **Testing:** Vitest (unit), RTL + MSW (integration), Playwright (E2E including `must_change_password` redirect gate).
