PRODUCT REQUIREMENTS DOCUMENT
# Orthodontic CRM Platform

Lead Management · Marketing Attribution · AI-Powered Engagement

Version 1.0  |  March 18, 2026  |  Confidential

## 1. Executive Summary

This document defines the product requirements for a proprietary browser-based CRM platform designed specifically for multi-location orthodontic practices. The CRM is the first phase of a two-system platform — it ships independently before the EHR and delivers standalone value by solving orthodontic marketing's most persistent problem: zero visibility into which marketing channels actually produce case starts.

The CRM is intentionally built without PHI at launch. Leads are prospective patients — their data (name, phone, email, treatment interest) is not Protected Health Information until linked to clinical records. This means the CRM operates outside HIPAA until EHR integration is added in a later phase, enabling faster development, lower compliance overhead, and earlier go-live.

#### Design Principles

- Browser-first: zero desktop installation. Works on any device — desktop, tablet, phone.
- Multi-location native: 34 locations from day one. Every feature designed for network-level management.
- Ortho-specific: not a generic CRM. Pipelines, stages, and workflows match actual ortho lead journeys.
- AI-augmented: Claude API powers intelligent lead response, follow-up drafting, and conversation assistance.
- Attribution-obsessed: every lead tagged with full UTM chain, call source, and referral origin.
- EHR-ready: data model and API designed for seamless future integration with the EHR platform.


#### What This CRM Is Not

> The CRM does not handle PHI, clinical records, treatment plans, billing, or insurance. Those belong to the EHR platform. The integration point is lead conversion: when a prospect signs a contract and becomes a patient, the CRM passes the record to the EHR and tracks attribution. Until that integration is built, conversion is tracked via Ortho2 CSV import.

## 3. Lead Capture & Attribution

Every lead entering the CRM must carry a complete attribution record — the full chain from ad platform to campaign to ad to keyword to landing page to form submission. This attribution data is the foundation of all marketing analytics.

### 3.1 Lead Capture Channels

| Channel                       | How It Works                                                                                                                    | Attribution Data Captured                                                                                  | Priority |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | -------- |

| Google Ads lead forms         | Google Ads API integration. Lead form extensions captured natively via webhook without requiring website visit.                 | Campaign ID, ad group ID, keyword, match type, placement, device, ad ID                                    | P1       |

| Call tracking (Twilio)        | Unique Twilio phone numbers provisioned per campaign/channel. Calls routed to practice line. Lead auto-created on call receipt. | Source phone number, campaign number dialed, call duration, recording (with consent disclosure), timestamp | P1       |
| Referral links                | Unique URL per referring doctor or patient. Click creates or updates a lead record with referral source.                        | Referrer ID, referrer type (doctor/patient), campaign tag, click timestamp                                 | P1       |

### 3.2 Attribution Data Model

Every lead record stores the following attribution fields. These fields are immutable after creation — first-touch attribution is locked. Last-touch attribution is tracked separately as stage events.

- `first_touch_source` — e.g. google, facebook, referral, direct, organic
- `first_touch_medium` — e.g. cpc, social, referral, none
- `first_touch_campaign` — campaign name from UTM or ad platform
- `first_touch_ad` — ad name or ID
- `first_touch_keyword` — search keyword (paid search only)
- `first_touch_landing_page` — full URL of first page visited
- `first_touch_referring_url` — URL that sent the visitor
- `first_touch_device` — mobile / desktop / tablet
- `call_tracking_number` — Twilio number that received the call (if call source)
- `referrer_id` — ID of referring doctor or patient (if referral source)
- `ad_platform_lead_id` — native lead ID from Google or Meta (if ad platform source)
- `created_at` — timestamp of lead creation
- `created_by_location` — location ID where lead was captured or assigned

### 3.3 Duplicate Detection & Merge

Leads are deduplicated on creation using phone number and email as primary keys. When a potential duplicate is detected:

- Exact phone OR email match: flag as duplicate, prompt coordinator to merge or keep separate
- Merge preserves all attribution data from both records — first touch from earliest record, last touch from most recent
- Merged records maintain full activity history from both originals
- Manual override available for coordinators to merge or unmerge


## 4. Nurture Automation

Automation sequences are triggered by lead stage changes, time elapsed, and lead behavior. All sequences are configurable per pipeline and per location. No sequence sends PHI — all clinical details stay off Twilio.

### 4.1 New Patient Pipeline — Automated Sequences

#### Sequence A: Immediate Response (triggers on lead creation)

| Step | Delay  | Channel | Message Type                                       | AI Assist                                         |
| ---- | ------ | ------- | -------------------------------------------------- | ------------------------------------------------- |
| 1    | 0 min  | SMS     | Welcome + exam booking link                        | AI personalizes based on treatment interest field |
| 2    | 2 hrs  | SMS     | Follow-up if no booking link clicked               | AI drafts based on lead source context            |
| 3    | 24 hrs | Email   | Free exam value proposition + patient testimonials | Template with dynamic name/location merge         |
| 4    | 3 days | SMS     | Final outreach before stage timeout                | AI draft — coordinator reviews before send        |
| 5    | 5 days | Task    | Assign manual call task to coordinator             | System notification only — no auto-send           |

#### Sequence B: Appointment Confirmation (triggers on Exam Scheduled)

| Step | Timing                                   | Channel | Content                                                       |
| ---- | ---------------------------------------- | ------- | ------------------------------------------------------------- |
| 1    | Immediately                              | SMS     | Appointment confirmed — date, time, location, directions link |
| 2    | 48 hrs before                            | SMS     | Reminder with rescheduling link                               |
| 3    | 2 hrs before                             | SMS     | Final reminder — 'We'll see you soon'                         |
| 4    | 30 min after scheduled time (if no-show) | Task    | Alert coordinator: patient did not arrive — follow up         |

#### Sequence C: Post-Exam Treatment Decision (triggers on Exam Completed)

| Step | Delay   | Channel | Content                                                              |
| ---- | ------- | ------- | -------------------------------------------------------------------- |
| 1    | 4 hrs   | SMS     | Thank-you + link to view treatment plan overview (no PHI)            |
| 2    | 24 hrs  | Email   | Treatment options overview + financing information + CareCredit link |
| 3    | 3 days  | SMS     | AI-generated check-in based on coordinator's exam notes summary      |
| 4    | 7 days  | Email   | Patient stories / before-after (with consent) + FAQ                  |
| 5    | 10 days | Task    | Coordinator call task: decision follow-up                            |
| 6    | 14 days | SMS     | Last chance — limited consultation availability messaging            |

### 4.2 In Retention Pipeline — Automated Sequences

- Post-treatment congratulations sequence: SMS day of completion + email with care instructions
- Retainer compliance reminders: weekly SMS for first month, monthly thereafter
- Recall reminders: 30-day, 7-day, and day-of SMS for scheduled recall appointments
- Referral ask: 90 days post-treatment — SMS with unique referral link
- Annual check-in: 12 months post-treatment — engagement and re-appointment nudge
- Birthday message: optional, configurable per location — drives goodwill and engagement


### 4.3 Sequence Configuration

Marketing managers can configure all sequences through the browser-based sequence builder:

- Add, remove, or reorder steps within any sequence
- Set delay timing (minutes, hours, days) per step
- Choose channel per step: SMS, email, task, or internal notification
- Enable or disable AI assistance per step
- Set sequence active hours (e.g. no auto-SMS before 8am or after 8pm)
- Configure opt-out handling — any STOP response removes lead from all sequences
- A/B test message variants — split traffic between two versions, track conversion
- Per-location overrides — locations can disable or modify sequences within bounds set by marketing manager

### 5.3 AI Agent Autonomous Mode (Configurable)

For marketing managers who want fully automated first-response handling, AI Agent mode can be enabled per sequence step. In Agent mode, Claude handles initial lead responses without coordinator involvement, with escalation rules for complex conversations.

- Agent mode only available for: first-response SMS (sequence step 1), appointment reminders, and recall reminders
- Agent automatically escalates to human when: patient asks a clinical question, patient expresses distress, conversation goes beyond 3 exchanges, or agent confidence is below threshold
- Full conversation transcript available to coordinator at any time
- Agent mode requires marketing manager activation — off by default
- All agent messages include compliant disclosure: 'This message was sent automatically. Reply STOP to opt out or call us at [number] to speak with our team.'
- Agent mode disabled immediately if coordinator manually sends a message in the thread — human takes over

## 7. Referral Tracking & Management

### 7.1 Patient Referral Program

Each patient who completes treatment receives a unique referral link. When a prospective patient clicks the link and submits a form, the referral chain is tracked end-to-end.

- Unique referral URL generated per patient at conversion: `yourpractice.com/ref/[unique-code]`
- Referral link included in post-treatment thank-you sequence automatically
- When a referred lead converts, the referring patient is credited in the system
- Referral rewards tracking: log whether a reward was issued (gift card, account credit, etc.) — CRM does not process payments, only logs the reward event
- Referring patient receives optional SMS notification when their referral books an exam and when they sign a contract
- Referral leaderboard available to staff: top referring patients by location

### 7.2 Referring Doctor Program

Orthodontic practices depend heavily on GP and pediatric dentist referrals. The CRM tracks every doctor referral with the same precision as paid advertising.

- Unique referral link per referring doctor practice
- Doctor referral portal — optional lightweight web view where referring doctors can see their referral status without logging into the full CRM
- Referring doctor dashboard: referrals sent, exams completed, cases started — all visible to the referring office
- Automated thank-you to referring doctor when their referral starts treatment
- Referral volume report: rank all referring doctors by case starts over any period
- Referring doctor contact management: store doctor name, practice, address, phone for relationship management

### 7.3 Referral Attribution in Analytics

Doctor and patient referrals appear as channels in the main analytics dashboard alongside Google and Facebook. Marketing managers can compare the cost per case start of referral programs against paid advertising — enabling data-driven decisions about where to invest in relationship marketing vs. digital advertising.

## 8. Email Marketing Campaigns

The CRM includes a full email marketing module for bulk campaigns to segmented lead and patient lists. Email campaigns are separate from the automated nurture sequences — they are one-time or recurring broadcasts to defined audiences.

### 8.1 Campaign Builder

- Drag-and-drop email designer — build HTML emails without coding
- Template library: pre-built templates for common campaigns (seasonal promotions, back to school, new patient specials, referral requests, retention check-ins)
- Brand configuration: logo, colors, fonts, signature configured once per location and applied to all campaigns
- Personalization tokens: `{{first_name}}`, `{{location_name}}`, `{{coordinator_name}}`, `{{referral_link}}` and custom fields
- Plain text fallback: automatically generated for deliverability
- Preview and send test: preview in browser, send test to email address before broadcast
- Subject line A/B testing: split audience between two subject lines, auto-select winner after 4 hours based on open rate

### 8.2 Audience Segmentation

Campaign recipients built from dynamic segment filters:

- Pipeline: New Patient / In Treatment / In Retention
- Stage: any combination of stages within a pipeline
- Location: one location, multiple locations, or all
- Lead source: only Google leads, only referrals, etc.
- Date range: leads created in last 30 days, or specific date range
- Treatment interest: braces, Invisalign, retainers, etc. (if captured on intake form)
- Last contact: no contact in X days
- Custom tag: any custom tag applied to leads
- Exclusions: exclude any segment from the send list (e.g. exclude anyone who already scheduled an exam)

### 8.3 Sending Infrastructure

- SendGrid API for all bulk email delivery — HIPAA-compliant with BAA not required (no PHI in emails)
- Dedicated sending domain per practice for deliverability — no shared sending IPs
- Automatic unsubscribe handling: CAN-SPAM compliant one-click unsubscribe in every email
- Bounce handling: hard bounces auto-removed from list, soft bounces retried
- Spam score preview: check email against spam filters before send
- Send time optimization: optional AI-powered send time selection based on historical open rates

### 8.4 Campaign Analytics

- Delivery rate, open rate, click rate, unsubscribe rate per campaign
- Click map: which links in the email were clicked most
- Conversion tracking: leads who opened email and then booked an exam or advanced pipeline stage within 7 days
- Campaign comparison: compare performance of multiple campaigns side by side
- Revenue attribution: if a lead who received a campaign converts, attribute a portion to that campaign


## 9. Ortho2 CSV Import & Pipeline Management

Until the EHR integration is built, the CRM connects to Ortho2 via manual CSV export and import. This is the mechanism for keeping pipeline membership accurate — removing converted patients from the New Patient pipeline and moving completed patients to Retention.

### 9.1 Import Types

| Import Type                         | Ortho2 Report to Export                         | CRM Action                                                                                                                                                    | Frequency                           |
| ----------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| Active patients — treatment started | Patients with status 'Active' or 'In Treatment' | Match on name + phone/email. Mark as 'Contract Signed' in New Patient pipeline. Create record in In Treatment pipeline. Remove from active nurture sequences. | Weekly — recommended Monday morning |
| Completed patients                  | Patients with status 'Debanded' or 'Completed'  | Match on name + phone/email. Mark treatment complete. Move to In Retention pipeline. Start retention nurture sequence.                                        | Weekly — recommended Monday morning |
| Scheduled appointments              | Upcoming appointment list                       | Match lead to appointment. Update stage to 'Exam Scheduled' if exam appointment. Log appointment details.                                                     | Daily — recommended morning         |
| No-shows                            | Appointments with no-show status                | Update CRM: mark exam as no-show. Trigger no-show re-engagement sequence. Move back to Contacted stage.                                                       | Daily                               |

### 9.2 Match Logic

CRM matches Ortho2 records to CRM leads using the following priority order:

1. Exact mobile phone match (highest confidence)
2. Exact email match
3. First name + last name + home phone match
4. First name + last name + date of birth match
5. No match found — flagged for manual coordinator review

### 9.3 Import Interface

- Simple web upload: coordinator drags CSV file into browser upload zone
- Column mapping: system auto-detects Ortho2 column names, shows mapping preview, allows manual correction
- Validation report: before committing, show count of matches, no-matches, and potential duplicates
- Preview mode: see exactly which records will be updated before confirming
- Import log: full history of all imports with user, timestamp, file name, and outcome counts
- Undo: bulk undo of the last import within 2 hours of execution


> When the EHR integration is built, this CSV import process becomes fully automated — the EHR sends conversion events directly to the CRM API in real time. The CSV import is a bridge, not a permanent architecture.

### 10.1 Multi-Location Access Control

Location scoping works as follows:

- Call Center Agents: see only leads assigned to their home location. Cannot view, edit, or contact leads at other locations.
- Call Center Managers: see all leads at their assigned location(s). Can be assigned to multiple locations.
- Marketing Staff: read-only view of all leads across all locations. Can create campaign drafts and sequence drafts for manager approval.
- Marketing Managers: full access across all 34 locations. Only role that can publish live automations, enable AI agent mode, manage staff accounts, and view system logs.
- Super Admin (developer-level): full system access including infrastructure configuration. Not a regular staff role.


## 11. Staff Portal

## 13. Future EHR Integration Design

The CRM is designed from day one to integrate cleanly with the EHR when it is ready. The following outlines the integration contract so both teams build toward compatibility. This section is for planning only — no integration code is built in Phase 1.

### 13.1 Integration Events (EHR → CRM)

| Event                  | Trigger                                     | Data Sent to CRM                                                    | CRM Action                                                            |
| ---------------------- | ------------------------------------------- | ------------------------------------------------------------------- | --------------------------------------------------------------------- |
| appointment_created    | Patient books exam in EHR                   | appointment_id, patient_id, date, time, location_id, provider_id    | Update lead stage to Exam Scheduled                                   |
| appointment_completed  | Appointment marked complete in EHR          | appointment_id, outcome (completed/no-show/rescheduled)             | Update lead stage based on outcome                                    |
| contract_signed        | Treatment contract signed in EHR            | patient_id, contract_value, treatment_type, start_date, location_id | Convert lead to patient, create In Treatment record, lock attribution |
| treatment_completed    | Patient debanded or treatment closed in EHR | patient_id, completion_date, treatment_duration                     | Move to In Retention pipeline, trigger retention sequence             |
| patient_record_created | New patient created in EHR                  | patient_id, name, dob, phone, email                                 | Link to existing CRM lead record by phone/email match                 |

### 13.2 Integration Events (CRM → EHR)

| Event                 | Trigger                            | Data Sent to EHR                                            | EHR Action                                                  |
| --------------------- | ---------------------------------- | ----------------------------------------------------------- | ----------------------------------------------------------- |
| lead_converted        | Contract signed event received     | lead_id, attribution_data, pipeline_history, coordinator_id | Attach attribution to patient record for lifetime reporting |
| appointment_requested | Lead clicks booking link from CRM  | lead contact info, preferred date/time, location            | Pre-fill new patient intake form in EHR                     |
| referral_source       | Any lead with referrer_id converts | referrer_id, referrer_type, lead_id, patient_id             | Log referral in EHR patient record for clinical context     |

### 13.3 Shared Data Infrastructure

- Same PostgreSQL cluster — CRM in `crm` schema, EHR in `pms` schema. Cross-schema queries available for reporting.
- Same Auth0/Supabase tenant — staff log in once, access both systems based on role.
- Same AWS VPC — internal service-to-service communication over private network, no public internet hop.
- Same S3 bucket structure — shared media storage with per-system prefix paths.
- EventBridge event bus — CRM and EHR publish events; each system subscribes to relevant events. Decoupled, no direct API dependency.
