## Codebase Patterns
- `insertActivity()` returns `string | null` (inserted id or null on conflict skip) and automatically updates `leads.last_activity_at` when a row is inserted
- Event worker creates its own `EventBus` instance; `subscribe()` must be called before `start()`
- Only `ad_lead.received` handler receives the `bus` as 3rd argument (for publishing `lead.created`); all other handlers receive `(event, db)` only
- Event type naming: `{domain}.{action}` (e.g., `lead.created`, `opt_out.received`)
- `CreateLeadData` requires ALL lead fields (except id/created_at/updated_at) â€” must explicitly set nullable fields to `null`
- Handler functions needing `bus` require `as Handler` cast in event-worker.ts since `Handler` type has `bus?` optional
- `calculateScore` expects `last_activity_at` as `Date | null` but `Lead.last_activity_at` is `string | null` â€” always convert with `new Date()` when passing to ScoreParams
- Handler pattern for stateful updates: findById â†’ db.transaction â†’ update fields â†’ findLastInboundAt â†’ calculateScore â†’ update score â†’ insertActivity
- Phone lookup handler pattern: normalizePhone (warn+return on fail) â†’ findByPhone (warn+return if empty) â†’ take `leads[0]` â†’ db.transaction with status/score/activity updates
- Cast `lead.contact_status as ContactStatus` when passing to applyOptOut/removeOptOut/applyHardBounce
