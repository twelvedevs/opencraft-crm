## Codebase Patterns
- Event types in @ortho/types follow: Payload interface + optional Event wrapper with `event_type`, `entity_type`, `entity_id`, `payload`
- Not all payloads have Event wrappers — some are payload-only (e.g. LeadStageChangedPayload)
- ROLE_PERMISSIONS in @ortho/auth-middleware uses `resource:action` format (e.g. `referrals:read`)
- super_admin bypasses permission checks in require-permission.ts (early return), but still has explicit permissions listed
- All services use ESM (`"type": "module"`) with NodeNext module resolution
- Migration naming: `YYYYMMDDhhmmss_description.ts` (campaign pattern); use `knex.schema.withSchema('schema_name').createTable()` for tables
- CHECK constraints and indexes added via `knex.raw()` after table creation
- Each service has its own `knexfile.ts`, `src/db.ts`, and `src/env.ts` — no shared @ortho/db package
- Referral service does NOT need bullmq/ioredis — uses @ortho/event-bus (SQS) only
- EventBus factory reads EVENT_BUS_DRIVER from process.env directly (no need to pass it)
- Cursor pagination: Base64 JSON `{ lastSeenId, lastSeenSortValue }`, tuple compare `(sort_col, id) < (?, ?)`, fetch `limit+1` then pop
- ON CONFLICT `.onConflict('col').ignore()` returns empty array on conflict — check `rows[0] ?? null`
- Repos export functions (not classes), accept `db: Knex` as first param, use `| null` for nullable fields
- Event publisher pattern: export async function taking `(bus: EventBus, payload: TypedPayload)`, call `bus.publish({ event_type, entity_type, entity_id, payload })`
- Inter-service HTTP clients use `X-Internal-Api-Key` header with service-specific API key env var
- Must build @ortho/types (`npm run build`) before dependent services can typecheck (dist/*.d.ts required)
- Services throw HTTP errors via `Object.assign(new Error(...), { statusCode: N })` — routes can read `.statusCode`
- Notification service uses direct `fetch()` to Messaging Service: POST /messages/send with `{ to, body, dedup_key }`
- SMS dedup_key pattern: `referral_{type}_notify:{referral_id}` — prevents duplicate SMS on SQS redelivery
- Aggregate counts in services use PostgreSQL `COUNT(*) FILTER (WHERE ...)::int` for single-query summaries
- Handlers that publish events take `(event, db, bus)` signature; read-only handlers take `(event, db)` only
- worker.ts follows campaign pattern: createEventBus() → subscribe → bus.start(); SIGTERM calls bus.stop() + destroy()
- Fastify 5 redirect: `reply.status(302).redirect(url)` — redirect() takes only URL string, set status separately
- Public routes: register via encapsulated `fastify.register(async (app) => { ... })` scope without authPlugin
- Staff routes: use `requirePermission('resource:action')` as `preHandler` array; access user ID via `req.user!.sub`
- Route registration: `app.register(routesFn, { prefix: '/path', db })` — route paths are relative to prefix
- index.ts pattern: two encapsulated scopes — public (no authPlugin) and staff (with authPlugin inside scope) — keeps auth isolated from public routes
- Leaderboard/aggregate queries: use `COUNT(*) FILTER (WHERE condition)::int` with parameterized bindings for safe period filtering
- Integration tests: mock external HTTP via `vi.mock()` at module level + `vi.stubGlobal('fetch', fetchMock)` — real Postgres, no HTTP calls
- Integration test HAS_DB guard: `describe.skipIf(!HAS_DB)` skips tests when DATABASE_URL is the fallback dummy value
- Route integration tests: use `buildTestApp()` + `app.inject()` — no port binding, JWT via test RSA keypair, JWKS mocked via `globalThis.fetch`
- Mock EventBus for handler tests: `{ published: [], publish(e) { this.published.push(e) }, subscribe() {}, start() {}, stop() {} }`
- Integration test helpers pattern: getDb(), runMigrations(), rollbackMigrations(), cleanup(), truncateTables(), and factory functions (insertReferrer, insertReferralLink, insertReferral)
