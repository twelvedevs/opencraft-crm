## Codebase Patterns
- ioredis named import: use `import { Redis } from 'ioredis'`; `vi.mock('ioredis', () => ({ Redis: vi.fn(() => mockRedis) }))` in tests
- ioredis constructor: pass `{ lazyConnect: true }` to avoid eager connection in tests / factory validation
- xpending has two forms: range (5 args: key, group, start, end, count) → delivery count per message; summary (2 args: key, group) → total pending count
- Differentiate xpending forms in mocks via `args.length === 5` check in `mockImplementation`
- Package structure mirrors `apps/platform/automation`: `type: module`, NodeNext module resolution, vitest for tests
- Use `.js` extensions in imports even for `.ts` source files (NodeNext requirement)
- `tsconfig.json` does NOT extend a workspace base — inline the full compilerOptions (no workspace tsconfig found)
- Tests live in `src/` alongside source files (e.g. `src/event-bus.test.ts`)
- Each package has its own `package-lock.json` (standalone installs, not a workspace with hoisting)
- Async poll loops must add `await new Promise(r => setTimeout(r, 0))` on empty-result branches to yield macrotasks; without it, immediately-resolving mocks starve timers in tests
- Always wrap poll loops in try/finally to guarantee stop-resolve fires even on unexpected errors
- When mocking `mockSend.mock.calls.find(...)`, cast via `(args[0] as { _type: string })._type` — destructured tuple annotations conflict with `any[][]` call types
