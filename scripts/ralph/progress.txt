## Codebase Patterns
- Services use a global `src/db.ts` exporting a knex singleton (`export default knex`); metrics-calculator and other services import it directly — do not pass db as a parameter to service-level functions that wrap analytics/KPI computation
- ROLE_PERMISSIONS in packages/@ortho/auth-middleware/src/permissions.ts is the single source of truth for all service permissions; add new `<service>:read` and `<service>:write` entries there for every new service
- Permission naming convention: `<service>:read` and `<service>:write` (e.g., reporting:read, reporting:write)
- `@ortho/db` does not exist as a package — use direct knex + pg pattern (same as referral/campaign services)
- vitest.config.ts: omit `setupFiles` if no integration setup file exists yet; add it later in the integration test story
- Migrations use `knex.schema.withSchema('crm_<service>').createTable(...)` for tables; use `knex.raw(...)` for CHECK constraints, custom indexes, and schema creation/drop
- Foreign keys referencing across schemas use full `'crm_reporting.table_name'` string in `.inTable()`
- `text[]` PostgreSQL arrays require `table.specificType('col', 'text[]')` in Knex — no native array method
