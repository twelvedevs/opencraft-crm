## Codebase Patterns
- For transactional multi-table inserts (e.g., rule + version), add a dedicated method to the repository that calls `this.db.transaction(async (trx) => { ... })` — keeps route handlers clean
- TypeBox `Type.Null()` as the 204 response schema allows `reply.status(204).send()` without TypeScript errors — omitting it causes TS2345 since Fastify infers reply types from the schema response map
- Knex db instance created via `createDb()` in `src/db.ts`; injected into Fastify plugins via plugin options (`FastifyPluginAsync<{ db: Knex }>`)
- Use `Type.Any()` for timestamptz date fields in TypeBox response schemas — avoids fast-json-stringify coercion issues with JS Date objects
- Fastify v5 route plugin pattern: `const plugin: FastifyPluginAsync<Opts> = async (fastify, opts) => {}; export default plugin`
- Always add `.gitignore` with `node_modules/` and `dist/` before staging service files — `git add apps/platform/<service>/` will otherwise pull in node_modules
- ESM throughout: `"type": "module"` in package.json, `module: NodeNext` + `moduleResolution: NodeNext` in tsconfig.json
- Fastify v5 with `@fastify/sensible`; use `logger: true` for pino — no pino-pretty dep needed
- `tsc --noEmit` is the typecheck command (via `npm run typecheck`)
- Vitest runs with no config file needed; `npm run test` (vitest run) works out of the box for ESM projects
- Knex migrations: use `withSchema('platform_automation')` on every `knex.schema` call; create schema in migration 001 with `knex.raw('CREATE SCHEMA IF NOT EXISTS platform_automation')`
- knexfile.ts lives in `migrations/` alongside migration files; set `directory: '.'` in migrations config
- tsconfig.json `include` only covers `src/**/*` — migration files are excluded from typecheck by design (avoids rootDir conflict)
- Distinguish "not found" (404) from "exists but deleted" (409) by calling `findByIdRaw` (no deleted filter) before the deleted-aware methods
- TypeBox Union with Literal values in the body schema validates status at the Fastify schema layer; an explicit status check in the handler is still harmless but redundant
- Test file imports use `.js` extension on relative paths (ESM NodeNext): `../../src/services/rule-validator.js`
- Branch depth starts at 1 for the root node; check `depth >= 4` to reject depth-4 branch nodes
- `send_message` missing `dedup_key` is a `console.warn` only — not added to errors array
