## Codebase Patterns
- Use `@sinclair/typebox` Type.Object + Value.Check for env validation in env.ts
- All services: `"type": "module"`, ESM with `.js` extensions in imports
- tsconfig: ES2022 target, NodeNext module/moduleResolution, strict: true
- Multi-stage Dockerfile: builder stage compiles TS, runner stage copies dist + node_modules
- Knex for migrations only; raw pg Pool + PoolClient for runtime queries (per updated spec)
- Migrations use `knex.raw()` with schema-qualified table names; first migration creates schema with `IF NOT EXISTS`
- Run `npm run typecheck` from within the service directory (no monorepo-level runner)
- google-ads-api npm package requires refresh_token (gRPC); for access_token use Google Ads REST API directly
- All platform services live under `apps/platform/<service-name>/`
- Workspace refs use `file:../../../packages/@ortho/<pkg>` format
- Repository pattern: standalone exported async functions with PoolClient as first arg, schema-qualified table names
- @ortho/types must be built (`npm run build` in packages/@ortho/types) before dependent services can typecheck against it
- ConnectorRegistry is a value (Map), not a type — use `typeof ConnectorRegistry` for parameter types in workers/services
- Connectors handle token decryption internally — callers of fetchSpend/fetchSpendRange don't need encryption keys
- ioredis: use `{ Redis }` named import (not default import) — consistent across all services
