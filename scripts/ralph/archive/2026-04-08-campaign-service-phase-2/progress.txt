## Codebase Patterns
- Integration tests for worker logic: simulate processJob by calling resolveAudience + orchestrateNonAB directly — never import worker module in tests (side-effect: creates BullMQ Worker)
- Integration test fetch mocking: compose JWKS mock (delegate to savedFetch) + service mocks in a single createServiceFetchMock helper; restore in afterEach
- HAS_REDIS guard: setup.ts uses `??` for BULLMQ_REDIS_URL; helpers.ts checks `!== 'redis://localhost:6379'` (mirrors HAS_DB pattern)
- BullMQ Queue for route handlers: create inside workflowRoutes() using `new Queue(ORCHESTRATE_QUEUE, { connection: bullmqRedis })` — avoids module-level side effects
- Queue.remove(jobId) for cancelling delayed jobs — wrap in try/catch since job may already be processing
- Campaign service uses `src/api.ts` (not `src/app.ts`) for the buildApp function per PRD spec
- Auth: use `authPlugin` from `@ortho/auth-middleware` with `jwksUrl` and `allowedPaths: ['/health']` — requires `IDENTITY_JWKS_URL` env var
- DB: use `knexLib` default import from `knex`, searchPath `['crm_campaigns', 'public']`
- Logger: `createLogger('crm-campaign')` cast as `unknown as FastifyBaseLogger` for Fastify
- All services export `destroy()` from db.ts for clean shutdown
- Migrations: use `knex.schema.withSchema('crm_campaigns')` for tables, `knex.raw()` for CHECK constraints and indexes
- knexfile.ts: uses `process.env['DATABASE_URL']` directly (not env.ts), with `schemaName: 'crm_campaigns'` for migration tracking
- FK references within same schema use fully qualified: `.references('id').inTable('crm_campaigns.campaigns')`
- State machine: use `validateTransition(status, action)` before any status change; use `targetStatus(status, action)` to get destination status
- Service layer: pure functions (no DB), exported from `src/services/campaign-service.ts`; routes call these before repo writes
- Routes: register as Fastify plugin `async function fooRoutes(app, opts: { db: Knex })`, registered with `{ prefix: '/campaigns', db }` in api.ts
- Auth: use `requirePermission('campaigns:write')` as preHandler for all campaign routes; `requirePermission('campaigns:manage')` for manager-only routes
- Integration tests: RSA keypair + JWKS mock on globalThis.fetch; `describe.skipIf(!HAS_DB)` to skip without Postgres; `pool: 'forks'` + `singleFork: true` in vitest config
- Test factories: use direct DB insert (bypassing routes) via `insertCampaign(db, overrides)` to set up specific statuses
- env.ts: all Phase 2 env vars already present (BULLMQ_REDIS_URL, SQS_QUEUE_URL, EVENTBRIDGE_BUS_NAME, AUDIENCE_ENGINE_URL, LEAD_SERVICE_URL, EMAIL_SERVICE_URL, TEMPLATE_SERVICE_URL)
- package.json: bullmq, ioredis already in dependencies; start:api, start:worker, dev:worker scripts already present
- Phase 2 stubs: POST /:id/schedule, DELETE /:id/schedule, POST /:id/send-now return 501 in workflow.ts — ready to be replaced
- BullMQ queue names: ORCHESTRATE_QUEUE = 'campaign:orchestrate', AB_WINNER_QUEUE = 'campaign:ab-winner-select' (prefixed to avoid collision)
- SQS consumer: startConsumer(db, bus) pattern — bus instance shared between subscribe+publish; bus created with createBus({ queueUrl: env.SQS_QUEUE_URL })
- Workers: place in `src/workers/` dir, import as side-effect in `src/worker.ts`; export job data interface (e.g. `OrchestrateJobData`)
- For COALESCE in knex updates: use `db.raw('COALESCE(col, now())')` directly in the update object
- Repositories: use `Omit<Interface, 'id'>` for insert data type; `Partial<Omit<Interface, 'id'>>` for update data type
- campaign_recipients has composite PK (campaign_id, lead_id) — no auto-generated id column
- batchInsert: `db.batchInsert(TABLE, rows, chunkSize)` — third arg is chunk size (1000)
- ioredis: import as `{ Redis }` (named export), not default — default fails TS constructable check
- Queue connection: `src/queue/connection.ts` exports `bullmqRedis`, `ORCHESTRATE_QUEUE`, `AB_WINNER_QUEUE`
- Worker entry: `src/worker.ts` — import workers here; use `bullmqRedis.quit()` for graceful shutdown
- HTTP service calls: use native `fetch` + `new URL()` for building paginated requests; mock via `globalThis.fetch = vi.fn(...)` in unit tests
- Audience Engine API: segments → POST /audiences/segments/:id/evaluate { snapshot_id, entities, done }; inline → POST /audiences/evaluate { snapshot_id, filter, entities, snapshot: true, done }; snapshot → GET /audiences/snapshots/:id { entity_ids }
- Lead Service API: paginated GET /leads?contact_status=active&limit=500&offset=N → { items }; by IDs GET /leads?ids=csv → { items }
- SQS consumer: `startConsumer(db, bus)` in `src/handlers/sqs-consumer.ts` — subscribes to events then calls `bus.start()`; bus created via `createEventBus()` in worker.ts
- Event handlers: place in `src/handlers/` dir; export pure async function taking (payload, db, bus) — easy to unit test with vi.mock() on repos
- Terminal status logic: all_failed→'failed'; has_completion+has_errors→'completed_with_errors'; else→'completed'
